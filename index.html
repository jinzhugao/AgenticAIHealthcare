Later

How students should think about healthcare agents

This section plays the role of the original workshop's later/advanced notes, but for healthcare. Students should not start with autonomy. They should start with narrow task definitions, explicit safety boundaries, and human review.

Design checklist

  1. Choose one narrow workflow, such as inbox triage or visit summarization.
  2. Define the user: clinician, student, scheduler, nurse, patient educator, or analyst.
  3. Specify trusted inputs and refuse to hallucinate missing data.
  4. Require structured output so a human can review the work quickly.
  5. Add escalation logic for emergencies or uncertainty.

Failure checklist

  1. Did the agent invent diagnoses, medications, or lab values?
  2. Did it miss an emergency red flag?
  3. Did it confuse patient-reported symptoms with clinician findings?
  4. Did it hide uncertainty instead of surfacing it?
  5. Did it skip the required handoff to a human reviewer?

Starter Kit

Minimal healthcare agent system prompt

You can start from this template and adapt it to a narrow healthcare workflow.

You are a healthcare workflow agent for a teaching environment. Your job: 1. Complete one narrow task. 2. Use only the information provided to you. 3. Never invent symptoms, diagnoses, labs, medications, or vital signs. 4. State missing or uncertain information explicitly. 5. Escalate emergencies or severe red flags immediately. 6. Produce output in a structured format for human review. If the task is high risk, stop and generate a handoff note instead of direct advice.

Prompts

Starter prompts for healthcare workshop experiments

These replace the original prompt examples with healthcare-specific ones that are safer and more relevant for your summer course.

Prompt 1

Patient inbox triage agent

Build a project called “Patient Inbox Triage” that reads incoming patient portal messages and routes them into emergency escalation, same-day clinical review, routine review, or administrative handling. Require explicit red-flag detection, missing-information reporting, and a patient-facing reply draft that does not diagnose or prescribe. Include app code, tests, sample data, docs, and a short failure-mode analysis. Keep it scoped for a course demo.
Prompt 2

Remote monitoring dashboard

Download or simulate home monitoring data for a chronic-care scenario such as hypertension, heart failure, or sleep monitoring. Build a small web dashboard that shows trends, highlights possible deterioration, and drafts clinician summaries. Include safe escalation logic and make clear that the dashboard is for review support rather than independent diagnosis.
Prompt 3

Medication reconciliation agent

Build an agent that compares hospital discharge medications, outpatient medications, and patient-reported medications. It should identify possible duplicates, omissions, and dose conflicts, then prepare a structured pharmacist review summary. It must not recommend starting, stopping, or changing medications on its own.
Prompt 4

Clinical coding support

Design an agent that reads a progress note and extracts evidence relevant to coding support. For every extracted condition, quote the exact phrase that supports it. Separate documented conditions from suspected ones, do not assign final billing codes automatically, and return a coder review checklist.
Prompt 5

Wearable sleep or apnea monitoring agent

Create an agent that reviews wearable sleep summaries, pulse trends, motion signals, and symptom notes to determine whether the pattern looks stable, worsening, or unclear. The output should be a trend summary, a review recommendation, urgent flags, and a patient-facing next-step draft. The agent must not diagnose sleep apnea or prescribe treatment.
Prompt 6

Choose your own healthcare workflow

Pick a workflow from clinical operations, education, patient communication, remote monitoring, public health, or quality improvement. If you want better results, specify: - the user - the workflow goal - the available data - the required structured output - the safety boundaries - the required escalation or review path
